## Issues Found

### Critical

**C-001** — `upsert_pending_edge` creates self-loop edge, not cross-node edge  
File: `src/context_os/graph/mutations.py:222`  
The Cypher MERGE uses `(a)-[r:DEPENDS_ON ...]->(a)` — both endpoints of the edge point to the
same node `a` (the Slack signal). The pending edge is intended to connect the Slack Signal node
to the unresolved GitHub Artifact node, but because the Artifact is not yet in the graph, the
implementation creates a self-referential edge instead of creating a placeholder target node or
storing the pending reference in a separate structure. The pending edge will never correctly link
to the GitHub node when it is later ingested.

### Important

**I-001** — `auth/middleware.py` falls back to unverified JWT decoding on any Clerk SDK exception  
File: `src/context_os/auth/middleware.py:72-75`  
When `clerk_client.authenticate_request()` raises any non-AuthError exception (e.g. network
timeout, SDK misconfiguration), the code silently falls back to `_decode_jwt_payload()` which
decodes without signature verification. This means a network hiccup during Clerk JWKS fetch could
allow any JWT (including forged ones) to pass authentication. The fallback should either re-raise
or explicitly limit to only dev-mode testing conditions.

**I-002** — `db/engine.py` uses deprecated `asyncio.get_event_loop()` in sync callback  
File: `src/context_os/db/engine.py:37`  
`asyncio.get_event_loop().run_until_complete()` is deprecated since Python 3.10 and raises
`DeprecationWarning` in 3.12. Under Python 3.12 strict mode, this call inside a SQLAlchemy
`connect` event (which fires on a sync thread) may not have a running event loop and will fail.
The pgvector codec registration should use `asyncio.get_running_loop()` or, better, use the
async-native registration path via the async engine's `connect` event.

**I-003** — T014 specifies `LangfuseSpanProcessor` but implementation uses OTLP exporter  
File: `src/context_os/observability/tracer.py:27-83`  
The task spec requires `LangfuseSpanProcessor()` from the Langfuse v3 SDK. The implementation
instead uses `OTLPSpanExporter` with HTTP Basic Auth (the Langfuse v4 integration pattern). This
is functionally equivalent for Langfuse v4 but differs from the spec. The CLAUDE.md also notes
"LangfuseSpanProcessor". If the Docker Compose Langfuse instance is v3, this will not work.

### Minor

**M-001** — `OAuthToken.metadata_` (Python name) maps to `metadata` column (DB name), but
`api/ingest.py:143` accesses it as `token_row.metadata_`. The naming is correct but unusual.
Add a comment to avoid future confusion.

**M-002** — `JiraNomalizer` (intentional typo per task spec note in jira/normalizer.py:24) is
documented as intentional but will cause confusion for future contributors. A class alias
`JiraNormalizer = JiraNomalizer` would improve discoverability without breaking the spec.

**M-003** — Migration `20260518_0001_initial_schema.py:118-122` creates the `embedding` column as
`sa.Text()` then alters it. This two-step workaround is fragile — it leaves the column nullable
even after the ALTER, and the `sa.Text()` placeholder type will not match the ORM model's
`Vector(768)` type annotation during Alembic autogenerate checks.
